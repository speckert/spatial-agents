Spatial Indexing

H3 Hexagonal Tile Pyramid

Multi-resolution spatial index inspired by raster image pyramids — applied to vector event streams.

H3 (developed by Uber, Apache 2.0) assigns every point on Earth to a hexagonal cell at configurable resolutions. Hexagons tile without gaps, have uniform adjacency (every cell has 6 neighbors), and nest across resolutions. Tiles are served as static JSON files — no special infrastructure required.

Resolution Hex Edge Length Use Case Temporal Bin Approx File Size
res 3 12.4 km Global overview — vessel/aircraft density heatmap 1 day ~2 KB per cell
res 4 4.7 km Regional view — shipping lane activity, flight corridors 1 hour ~5 KB per cell
res 5 1.8 km Port/airport approach — individual track clusters 5 min ~10 KB per cell
res 6 0.68 km Harbor/runway detail — individual vessel/aircraft tracks 1 min ~15 KB per cell
res 7 0.26 km Berth/gate level — full kinematic state per entity live stream ~20 KB per cell
Python
import h3 from spatial_agents.spatial import tile_builder # Assign vessel position to H3 cells at multiple resolutions lat, lng = 37.8044, -122.2712 # Oakland outer harbor cells = { res: h3.latlng_to_cell(lat, lng, res) for res in range(3, 8) } # {3: '832830fffffffff', 4: '842831dffffffff', ...} # Generate tile file for a single cell tile_builder.build_tile( cell_id=cells[5], resolution=5, temporal_bin="5min", output_dir="/tiles/h3" ) # Writes: /tiles/h3/5/842831dffffffff.json

Design note: This multi-resolution tiling approach applies the same concept of level-of-detail hierarchies used in raster image pyramids, adapted here for vector event streams rather than pixels.

On-Device Intelligence

Foundation Models Evaluation Pipeline

The Python FM SDK (macOS) enables iterative prompt development, token budget analysis, and structured output validation — before shipping prompts in the Swift app.

Token Budget Management

4,096-token context window. System prompt + tool schemas + data payload + response must all fit. Python harness measures each component's token cost using tokenCount(for:) and alerts when budgets are exceeded.

Python SDK

Prompt Version Control

Each data domain (maritime, aviation, orbital) has versioned prompt templates. The eval harness runs regression suites when prompts change — ensuring the updated model in 26.4 still produces correct structured output.

Python + Swift

Schema Validation

FM guided generation outputs are validated against the same @Generable struct definitions used in Swift. Python-side validation catches schema drift before it reaches production clients.

Python + Swift

Context Window Strategy

Large geospatial datasets are compressed into context-budget payloads: statistical summaries, top-N anomalies, and causal graph excerpts — not raw position lists. The FM narrates intelligence, not data.

Causal Intelligence

Structural Causal Models

Pearl's SCM framework applied to geospatial event streams. The Python layer builds causal graphs from observed patterns; the on-device FM narrates them into actionable intelligence.

Know the How, Show the Why — moving beyond correlation dashboards to causal reasoning about why geospatial patterns emerge.

1

Event Detection

Identify behavioral patterns across data streams — vessel loitering, flight diversions, correlated AIS/ADS-B anomalies, dark vessel gaps.

2

DAG Construction

Build directed acyclic graphs encoding causal relationships: weather → route deviation → port congestion → vessel loitering. Nodes are observable events; edges encode causal direction.

3

Intervention Modeling

Pearl's do-calculus: "What would happen to port congestion if we do(reroute_traffic)?" Counterfactual reasoning over the causal graph without running real-world experiments.

4

Graph Serialization

DAG exported as JSON — nodes, edges, intervention results, confidence scores. Compact enough to fit within the FM's context window alongside the prompt template.

5

On-Device Narration

The Swift client passes the causal graph to the on-device FM, which generates human-readable intelligence: "Vessel loitering near Oakland increased 40% because storm-driven route changes created a 12-hour backlog."

Scalability Path

Phase 1 — Neural Magician

M1 Mac Mini (16 GB), macOS Tahoe 26.4. Apache + Certbot on ports 80/443, FastAPI on localhost:8012 behind reverse proxy. Handles thousands of concurrent clients. ~7W idle draw, runs headless indefinitely.

Current Target

Phase 2 — S3 Mirror

Pipeline writes tiles to local disk and syncs to S3 bucket. Static tile requests shift to S3/CDN for geographic distribution and uptime guarantees. API stays on the Mini.

When Needed

Phase 3 — Full Cloud

Pipeline containerized (ARM64). Deployed to cloud compute. S3 for tiles, managed API gateway. Triggered when subscriber count demands reliability SLAs, not by data volume.
